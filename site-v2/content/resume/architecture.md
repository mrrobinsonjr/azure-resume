## Azure Infrastructure Architecture — This Site

This resume site is built on a fully serverless, enterprise-grade Microsoft cloud stack. The architecture was intentionally designed to demonstrate hands-on experience with Azure services commonly used in large-scale government and commercial environments — the same patterns that power production systems at DoD IL4/IL5/IL6 compliance levels.

## Why Azure Static Web Apps (SWA) for Hosting

Azure SWA provides a globally-distributed, edge-cached static site without managing servers or load balancers. It handles HTTPS termination, CDN caching, and CI/CD from GitHub with zero infrastructure code. For a resume site that sees sporadic traffic spikes (recruiter review sessions), this means fast page loads everywhere without paying for idle capacity — the same cost model used for internal enterprise documentation portals at scale.

## Azure Functions as API Layer

The chat assistant, visitor counter, and content ingestion APIs run on Azure Functions — a serverless compute platform that scales from zero to thousands of concurrent requests automatically. Each function is deployed independently with its own resource allocation and security boundary: the chat endpoint uses managed identity for secure OpenAI access, while the visitor counter writes only to Table Storage with no inbound network exposure. This separation mirrors enterprise microservice patterns where each capability has a distinct threat model.

## Azure Table Storage for Persistent State

The visitor counter uses Azure Table Storage — a low-cost, NoSQL key-value store that scales linearly without database administration. For this use case (a single integer counter updated thousands of times daily), Table Storage is dramatically more cost-effective than Cosmos DB or SQL Database while still offering the same 99.99% availability SLA and encryption at rest. It's the kind of right-sizing decision that enterprise architects make when balancing features against total cost of ownership.

## Azure OpenAI with RAG for Intelligent Chat

The "Ask about my experience" assistant uses Retrieval-Augmented Generation — not a raw LLM call, but a pipeline where user questions are first semantically matched against indexed content (roles, education, certifications) using vector embeddings generated by Azure OpenAI's text-embedding model. Only the most relevant chunks are sent to GPT for answer generation, which means responses cite specific roles and avoid hallucination. The embedding vectors are pre-computed as static artifacts in this repository — a deliberate choice that eliminates runtime latency and cost while keeping the chat experience fast even under heavy recruiter use during hiring cycles.

## Content Pipeline: Markdown → RAG Chunks → Embeddings

Content lives as human-readable markdown files (`site-v2/content/roles/*.md`, `summary.md`, etc.). The build pipeline (`scripts/build-rag.mjs`) parses these into structured chunks with metadata, then a separate batch script generates vector embeddings against Azure OpenAI — handling rate limits (20 chunks per request), DNS workarounds for corporate network environments, and shell quoting safely. This separation of content authoring from ML processing mirrors how enterprise teams manage data pipelines: writers produce markdown, engineers maintain the indexing infrastructure.

## Security and Compliance Posture

Every component uses Azure Managed Identities — no API keys stored in source control or environment variables at runtime. The Functions API is exposed only through SWA's built-in proxy (no public endpoint), and all storage accounts use private endpoints with network restrictions. This zero-trust architecture, where services authenticate to each other via managed identity rather than shared secrets, is the standard pattern recommended by Microsoft for production Azure deployments — including those subject to DoD IL5/IL6 compliance requirements.

## Why This Matters for Enterprise Recruiters

This stack demonstrates breadth across four categories of cloud engineering: infrastructure (SWA + Functions), data storage (Table Storage), AI/ML integration (OpenAI embeddings + RAG), and security/compliance (managed identity, private endpoints). Each choice was made with an understanding of trade-offs — cost vs. features, latency vs. complexity, security vs. convenience — which is exactly the decision-making that senior cloud architects perform daily in regulated enterprise environments.
